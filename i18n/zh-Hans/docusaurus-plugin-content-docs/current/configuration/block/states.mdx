## 简介

在 Minecraft 中，每个方块都可以拥有一个或多个**方块状态**。例如，原木有不同朝向（`axis=x/y/z`），树叶可以检测与树木的"距离"（`distance=1~7`）。这些状态共同决定了方块在游戏中的外观和物理行为。

在 CraftEngine 中，我们需要区分两个核心概念：

#### 🎨 视觉方块状态（客户端看到的）

这是玩家在客户端**实际看到**的方块状态，因此也叫**原版方块状态**。

- 插件通过为特定的原版状态（例如某个音符盒）附加自定义模型，来实现各种独特的方块渲染
- 简单来说：**视觉状态 = "看起来是什么样子"**

#### ⚙️ 内部方块状态（服务器端运算的）

这是在服务器端**真正被存储和运算**的方块状态，因此也叫**服务端方块状态**。

- 它负责处理方块的逻辑与物理行为——碰撞箱、红石信号、实体交互等
- 服务器将内部状态通过特定的网络协议，"映射"成一个视觉状态，再发送给客户端渲染
- 简单来说：**内部状态 = "实际如何运作"**

---

`state`/`states` 配置节做一件事：**为每个内部方块状态指定它在客户端显示为什么样子**。

根据方块的复杂度，有两种配置模式：

| 模式 | 配置键 | 适用场景 |
|:---|:---|:---|
| 单状态 | `state` | 方块只有一种内部状态（如灯笼、装饰品） |
| 多状态 | `states` | 方块有多种内部状态（如原木有朝向、树叶有距离/含水） |

下面从最简单的单状态开始，逐步深入到多状态。

## 单状态方块

当方块只有一种内部状态时，使用 `state` 配置节。它的核心就是 **视觉状态** + **模型** 的组合：视觉状态决定"用哪个原版方块作为载体"，模型决定"渲染成什么样子"。

```yaml
state:
  auto_state: note_block
  model:
    path: "minecraft:block/custom/my_block"
```

`auto_state` 让插件自动挑选一个音符盒状态，`model` 指定渲染的模型。下面逐一详解视觉状态和模型配置。

### 视觉状态详解

#### `auto_state`：让插件自动分配

```yaml
auto_state: note_block
```

插件会从预设的**状态组**中自动挑选一个空闲的原版状态。同一组内的所有状态共享关键特性（碰撞箱、渲染属性），你无需关心具体用了哪一个。

:::info
可用的 auto_state 组

| 组名 | 包含的原版方块 | 说明 |
|:---|:---|:---|
| `solid` | 音符盒 + 蘑菇方块 | 最通用的实心方块组 |
| `note_block` | 音符盒 | 常用，碰撞箱为完整方块 |
| `mushroom_stem` | 蘑菇柄 | 完整方块 |
| `red_mushroom_block` | 红色蘑菇方块 | 完整方块 |
| `brown_mushroom_block` | 棕色蘑菇方块 | 完整方块 |
| `mushroom` | 以上三种蘑菇方块 | 任意蘑菇方块 |
| `tintable_leaves` | 橡木/丛林/金合欢/深色橡木/红树 | 可着色树叶，不含水 |
| `waterlogged_tintable_leaves` | 同上 | 可着色树叶，含水 |
| `non_tintable_leaves` | 云杉/白桦/樱花/苍白橡木/杜鹃 | 不可着色树叶，不含水 |
| `waterlogged_non_tintable_leaves` | 同上 | 不可着色树叶，含水 |
| `leaves` | 所有树叶 | 不含水 |
| `waterlogged_leaves` | 所有树叶 | 含水 |
| `lower_tripwire` | 绊线（attached=true） | 碰撞箱较小 |
| `higher_tripwire` | 绊线（attached=false） | 碰撞箱较大 |
| `tripwire` | 绊线 | 任意绊线 |
| `sapling` | 所有树苗 | — |
| `pressure_plate` | 测重压力板 | — |
| `cactus` / `sugar_cane` / `weeping_vine` / `twisting_vine` / `cave_vine` / `kelp` / `chorus` | 对应植物 | — |

:::

**展开写法：共享视觉状态**

多个外观需要共用**同一个**原版状态时，使用展开写法：

```yaml
# 简单写法 — 各自独立分配不同的视觉状态
auto_state: solid

# 展开写法 — 相同 id 的外观共用同一个视觉状态
auto_state:
  type: solid
  id: "my_shared_id"
```

不指定 `id` 时，每个 `auto_state: solid` 各自独立分配。指定了相同 `id` 的，会被绑定到**同一个**原版方块状态。

#### `state`：精确指定原版状态

当你需要精确控制使用哪个原版状态时，使用 `state`。

```yaml
# 方式一：完整原版标识符
state: "minecraft:note_block[instrument=hat,note=0,powered=false]"

# 方式二：映射简写（需 block_state_mappings 预先配置）
state: "note_block:0"
```

:::caution
同一个原版方块状态**只能绑定一个**自定义模型。如果你把同一个原版状态绑给了两个不同的自定义模型，插件会在控制台打印冲突警告。
:::

### 模型配置

`model` 决定视觉状态在客户端渲染成什么样子。它有几种互斥的配置组合：

<details>
<summary>① 仅 path — 已有现成模型文件</summary>

```yaml
model:
  path: "minecraft:block/custom/my_block"
```

</details>

<details>
<summary>② path + generation — 让插件自动生成模型</summary>

```yaml
model:
  path: "minecraft:block/custom/my_block"
  generation:
    parent: "minecraft:block/cube_column"
    textures:
      end: "minecraft:block/custom/top"
      side: "minecraft:block/custom/side"
```

</details>

<details>
<summary>③ path + textures — 只关心贴图，插件推断模型结构</summary>

```yaml
model:
  path: "minecraft:block/custom/my_block"
  textures:
    - "minecraft:block/custom/bottom"
    - "minecraft:block/custom/top"
    - "minecraft:block/custom/side"
```

</details>

<details>
<summary>④ 仅 texture — 单张贴图，最简写法</summary>

```yaml
model:
  texture: "minecraft:block/custom/my_block"
```

等价的写法：

```yaml
# 以上等价于
model:
  textures: "minecraft:block/custom/my_block"

# 也等价于
model:
  textures:
    - "minecraft:block/custom/my_block"
```

单张贴图时 `path` 可省略（自动取贴图路径），`texture` 和 `textures` 等价，列表和字符串也等价。

</details>

<details>
<summary>⑤ models 列表 — 多个模型加权随机</summary>

```yaml
models:
  - path: "minecraft:block/custom/flower_1"
    weight: 8
  - path: "minecraft:block/custom/flower_2"
    weight: 5
```

</details>

<details>
<summary>⑥ transparent — 移除原始模型</summary>

用于移除方块的原始模型，通常用于空出状态以供方块实体渲染器使用。

```yaml
transparent: true
```

</details>

**`model` 的所有子项：**

| 子项 | 类型 | 说明 |
|:---|:---|:---|
| `path` | 字符串 | 模型文件路径。使用 `textures`（2 张及以上）或 `generation` 时**必须**填写，用于指定生成的模型存放位置 |
| `textures` / `texture` | 列表或字符串 | 简化贴图配置，根据数量自动推断模型结构（与 `generation` 互斥）。两个键名等价；1 张贴图可用字符串简写 |
| `generation` | 对象 | 让插件自动生成模型，需指定 `parent` 和 `textures`（与 `textures` 列表互斥） |
| `x` / `y` / `z` | 整数 | 绕轴旋转角度，必须为 90 的倍数（`z` 需要 1.21.11+） |
| `uvlock` | 布尔 | 是否锁定贴图方向，默认 `false` |
| `weight` | 整数 | 仅在 `models` 列表中使用，选中权重，默认 `1` |

**`textures` 的自动推断规则：**

| 贴图数量 | 推断模型 | `path` 是否必填 | 贴图顺序 | 范例方块 |
|:---:|:---|:---:|:---|:---|
| 1 | `cube_all` | 否（默认取贴图路径） | 六面相同 | 下界岩 |
| 2 | `cube_column` | **是** | 端面, 侧面 | 原木 |
| 3 | `cube_bottom_top` | **是** | 底面, 顶面, 侧面 | 草方块 |
| 4 | `orientable` | **是** | 底面, 顶面, 前面, 侧面 | 熔炉 |
| 5+ | 五面独立 | **是** | 底面, 顶面, 北, 东, 南, 西 | 合成器 |

> 仅 1 张贴图时 `path` 可省略，此时模型路径自动取贴图路径。2 张及以上**必须**显式指定 `path`，用于确定生成的模型文件存放位置。

```yaml
# 1 张贴图 — path 可省略，字符串简写
model:
  texture: "minecraft:block/custom/my_block"   # 单张贴图可直接写字符串

# 等价于
model:
  textures:
    - "minecraft:block/custom/my_block"

# 2 张贴图 — path 必填
model:
  path: "minecraft:block/custom/my_block"       # 必须指定模型路径
  texture:
    - "minecraft:block/custom/my_block_bottom"
    - "minecraft:block/custom/my_block_top"
```

贴图前加 `^` 前缀表示将该贴图同时用作粒子纹理（破坏方块时产生的粒子效果）：

```yaml
textures:
  - "^minecraft:block/custom/top"     # 顶面贴图 → 同时作为 particle
  - "minecraft:block/custom/bottom"
  - "minecraft:block/custom/side"
```

:::tip
`textures` vs `generation`

两者都在 `model` 下，互斥：

| | `textures`（列表） | `generation`（对象） |
|:---|:---|:---|
| 写法 | `model` 下配 `path` + `textures` | `model` 下配 `path` + `generation` |
| 适用 | 简单方块，只需贴图 | 需要特定 `parent`（如 `leaves`、`cross`、`stairs`） |
| `path` | 1 张贴图可省略，2+ 张必填 | 必填 |

```yaml
# textures 列表（自动推断模型结构）
model:
  path: "minecraft:block/custom/my_block"
  textures:
    - "minecraft:block/custom/my_block_bottom"
    - "minecraft:block/custom/my_block_top"
    - "minecraft:block/custom/my_block_side"

# generation 对象（手动指定父模型）
model:
  path: "minecraft:block/custom/my_leaves"
  generation:
    parent: "minecraft:block/leaves"
    textures:
      all: "minecraft:block/custom/my_leaves"
```
:::

## 多状态方块

当方块需要多种内部状态时，使用 `states` 配置节。

:::tip
何时需要多状态？
- 方块有**朝向**（原木、柱子、楼梯……）
- 方块可以**含水**（`waterlogged`）
- 方块需要**多种变体**（树叶的 distance/persistent）
- 方块需要**红石互动**（`powered`、`open`）
:::

`states` 节由三个子节组成，按顺序配置：

```
states:
  properties:   # ① 定义内部属性
  appearances:  # ② 定义可用的视觉外观
  variants:     # ③ 将内部状态映射到外观（可选）
```

### 渐进示例

以下用原木方块为例，展示从最简到完整的配置过程。

<details>
<summary>① 最简：只定义属性，不定义外观（用默认外观）</summary>

```yaml
blocks:
  default:palm_log:
    states:
      properties:
        axis:
          type: axis
          default: "y"
      appearances:
        default:
          auto_state: solid
          model:
            path: "minecraft:block/custom/palm_log"
```

只有 `default` 一个外观。由于插件在未匹配到其他外观时会使用第一个外观作为兜底，所以 `axis=x`、`axis=y`、`axis=z` 三种内部状态会**全部使用这个 default 外观**。同时因为属性名是 `axis`（特殊名称），插件会自动处理放置方向。

</details>

<details>
<summary>② 为不同方向配置不同模型</summary>

```yaml
blocks:
  default:palm_log:
    states:
      properties:
        axis:
          type: axis
          default: "y"
      appearances:
        axisY:
          auto_state: solid
          model:
            path: "minecraft:block/custom/palm_log"
            generation:
              parent: "minecraft:block/cube_column"
              textures:
                end: "minecraft:block/custom/palm_log_top"
                side: "minecraft:block/custom/palm_log"
        axisX:
          auto_state: solid
          model:
            x: 90
            y: 90
            path: "minecraft:block/custom/palm_log_horizontal"
            generation:
              parent: "minecraft:block/cube_column_horizontal"
              textures:
                end: "minecraft:block/custom/palm_log_top"
                side: "minecraft:block/custom/palm_log"
        axisZ:
          auto_state: solid
          model:
            x: 90
            path: "minecraft:block/custom/palm_log_horizontal"
            generation:
              parent: "minecraft:block/cube_column_horizontal"
              textures:
                end: "minecraft:block/custom/palm_log_top"
                side: "minecraft:block/custom/palm_log"
      variants:
        axis=y:
          appearance: axisY
        axis=x:
          appearance: axisX
        axis=z:
          appearance: axisZ
```

三个方向分别对应三个外观。通过 `variants` 将 `axis=y` 映射到竖版模型、`axis=x` 和 `axis=z` 映射到水平版模型（旋转角度不同）。

</details>

<details>
<summary>③ 用 textures 简化写法</summary>

```yaml
blocks:
  default:palm_log:
    states:
      properties:
        axis:
          type: axis
          default: "y"
      appearances:
        axisY:
          auto_state: solid
          model:
            path: "minecraft:block/custom/palm_log"
            textures:
              - "minecraft:block/custom/palm_log_top"
              - "minecraft:block/custom/palm_log_side"
        axisX:
          auto_state: solid
          model:
            x: 90
            y: 90
            path: "minecraft:block/custom/palm_log_horizontal"
            generation:
              parent: "minecraft:block/cube_column_horizontal"
              textures:
                end: "minecraft:block/custom/palm_log_top"
                side: "minecraft:block/custom/palm_log"
        axisZ:
          auto_state: solid
          model:
            x: 90
            path: "minecraft:block/custom/palm_log_horizontal"
      variants:
        axis=y:
          appearance: axisY
        axis=x:
          appearance: axisX
        axis=z:
          appearance: axisZ
```

`axisY` 外观使用 `textures` 简化写法（2 张贴图 → 插件推断 `cube_bottom_top`），无需手写 `generation`。

注意：`axisX` 和 `axisZ` 需要旋转 + 不同的父模型，`textures` 推断的模型不适用，所以仍需用 `model` + `generation`。

</details>

### 属性详解

属性定义了方块所有可能的内部状态。插件会计算所有属性的**笛卡尔积**，自动生成全部内部状态。

```yaml
properties:
  waterlogged:
    type: boolean          # true / false
    default: false
  distance:
    type: int              # 整数范围
    default: 7
    range: 1~7             # 最小值~最大值
  facing:
    type: horizontal_direction  # north / south / west / east
    default: north
```

每种 `type` 对应固定的可能值集合。例如 `boolean` = `{true, false}`，`horizontal_direction` = `{north, south, west, east}`。所有属性值的笛卡尔积就是全部可能状态。

> 例如：`waterlogged`(2) × `distance`(7) = 14 个内部状态

完整属性类型列表见 [ℹ️ 属性](states/properties.mdx) 子页面。

:::tip
特殊属性名
部分属性名会被插件硬编码特殊的放置行为：

| 属性名 | 触发的行为 |
|:---|:---|
| `axis` | 放置时自动对齐玩家朝向的轴向 |
| `facing` | 放置时自动对齐玩家朝向（6 方向） |
| `facing_clockwise` | 放置时自动对齐玩家朝向（4 方向，旋转 90°） |
| `rotation` | 精确旋转控制（`type: int`, `range: 0~7` 或 `0~15`） |
| `waterlogged` | 决定方块是否含水 |

如果不使用这些特殊名称（如 `custom_axis`），则不会触发任何自动旋转——方块始终使用默认值。
:::

### 外观详解

`appearances` 定义方块所有可能的**视觉外观**。每个外观内部的配置规则与单状态方块完全一致：`auto_state`/`state` + `model`（其下可选 `path`、`textures`、`generation` 等）/ `transparent` / `entity_renderer`。

外观名称自由定义，只需同一方块内不重复。**第一个定义的外观是"默认外观"**——当某个内部状态未匹配到任何 variant 的 `appearance` 时，自动使用它。

```yaml
appearances:
  任意名称A:        # ← 第一个 = 默认外观
    auto_state: solid
    model: ...
  任意名称B:
    auto_state: solid
    model: ...
```

### 变体匹配规则

`variants` 将内部状态映射到视觉外观。它的核心是**变体键**的匹配逻辑。

#### 变体键格式

```yaml
variants:
  waterlogged=false:           # 只约束一个属性
    appearance: default
  waterlogged=true:            # 约束一个属性 + 覆写设置
    appearance: waterlogged
    settings:
      resistance: 1200.0
      fluid_state: water
  facing=up:                   # 约束一个属性，只覆写设置不改外观
    settings:
      luminance: 15
```

变体键 = `属性名=值` 对，多属性用逗号分隔，**排列顺序不影响匹配**。

#### 三条核心匹配规则

**规则 1 — 通配符语义**

变体键中**未列出的属性** = 匹配该属性的**所有可能值**。

| 变体键 | 实际匹配 |
|:---|:---|
| `waterlogged=false` | 全部 `waterlogged=false`，**不论** persistent、distance 为何值 |
| `facing=up` | 全部 `facing=up`，其他属性不限 |
| `axis=y` | 全部 `axis=y`，其他属性不限 |

> 写哪些属性就约束哪些；没写的不限制。利用这个特性可以用极少条目覆盖所有状态。

**规则 2 — 多条目叠加**

同一个内部状态匹配多个 variant 条目时：
- `appearance`：取**第一个匹配条目**的值（后续不覆写）
- `settings`：所有匹配条目的 settings **合并叠加**，同名项后定义覆写先定义

**规则 3 — 默认兜底**

未匹配到任何带 `appearance` 的 variant 的状态 → 使用 `appearances` 下**第一个外观**。

#### 完整示例：树叶

以树叶方块为例，它有 3 个属性：`waterlogged`(2) × `persistent`(2) × `distance`(7) = **28 个内部状态**。

<details>
<summary>展开完整配置</summary>

```yaml
blocks:
  default:palm_leaves:
    states:
      properties:
        waterlogged:
          type: boolean
          default: false
        persistent:
          type: boolean
          default: true
        distance:
          type: int
          default: 7
          range: 1~7

      appearances:
        default:                          # 第一个外观 → 默认兜底
          auto_state: leaves
          model:
            path: "minecraft:block/custom/palm_leaves"  # 1张贴图可省略path，此处显式写出
            textures:
              - "minecraft:block/custom/palm_leaves"
        waterlogged:
          auto_state: waterlogged_leaves
          model:
            path: minecraft:block/custom/palm_leaves

      variants:
        waterlogged=false:                # → 覆盖 14 个状态
          appearance: default
        waterlogged=true:                 # → 覆盖 14 个状态
          appearance: waterlogged
          settings:
            resistance: 1200.0
            burnable: false
            fluid_state: water
```

</details>

**匹配过程演示** — 以内部状态 `waterlogged=true, distance=7, persistent=true` 为例：

```
遍历 variants：
  waterlogged=false             → ❌ 不匹配（waterlogged 值不同）
  waterlogged=true              → ✅ 匹配 → appearance=waterlogged
                                 → 继承 settings: {resistance:1200, burnable:false, fluid_state:water}

最终：
  外观 = waterlogged
  settings = {resistance:1200, burnable:false, fluid_state:water}
```

`waterlogged=false` 和 `waterlogged=true` 两个条目各自覆盖 14 个内部状态，合计覆盖全部 28 个——利用通配符语义，仅 2 个 variant 就能完成映射。

<details>
<summary>更多 variants 写法示例</summary>

**示例 A：用单个 variant 约束多个属性**

```yaml
variants:
  waterlogged=true,facing=north:
    appearance: waterlogged_north
```

只匹配同时满足 `waterlogged=true` **且** `facing=north` 的内部状态。

**示例 B：只覆写 settings，不动外观**

```yaml
variants:
  powered=true:
    settings:
      luminance: 7
```

所有 `powered=true` 的状态发光等级变为 7，外观沿用默认兜底。

**示例 C：同一状态匹配多个 variant**

```yaml
variants:
  waterlogged=true:
    settings:
      resistance: 100.0
      fluid_state: water
  facing=up:
    settings:
      resistance: 50.0     # ← 覆写前面的 100.0，最终 resistance=50.0
      luminance: 15        # ← 新增，最终 luminance=15
```

如果某个内部状态同时满足 `waterlogged=true` 和 `facing=up`，它的最终 settings 为：
`{resistance:50.0, fluid_state:water, luminance:15}`（`resistance` 被第二个条目覆写）。

### 内部状态数量速查

配置多状态方块时，了解变体总数有助于预估 ID 占用：

```
变体总数 = ∏ 每个属性的 possibleValues 数量

boolean        → 2
int(range)     → max - min + 1
axis           → 3
direction      → 6
horizontal_dir → 4
half / hinge   → 2
slab_type      → 3
stairs_shape   → 5
```

> 例如 `waterlogged(2) × persistent(2) × distance(7) = 28 个变体`

## 配置内部方块 ID

通常不需要手动配置。仅当你需要**固定内部 ID**（如配合数据包使用）时才需要。

```yaml
# 单状态：直接指定
state:
  id: 0
  auto_state: note_block
  model: ...

# 多状态：指定起始 ID，后续变体自动占连续区间
states:
  id: 100      # 28 个变体 → 占用 100~127
  properties: ...
```

- ID 必须**全局唯一**
- 不能超出 `config.yml` 中 `serverside-blocks` 的上限
